## Read This First ‚Äî Project Constitution + Principle Docs

This project is governed by a constitution and four detail-principle documents. **Every task in this repository ‚Äî from a one-line fix to a multi-file refactor ‚Äî must be bracketed by reading the relevant docs at the start and verifying against them at the end.** The reading is not optional; it is the project's enforcement mechanism for consistency across contributors (human or AI).

### The five canonical documents

| Document | When to consult |
| --- | --- |
| [`specs/constitution.md`](specs/constitution.md) | Always, before starting any non-trivial task. Eight Core Principles, the Definition of Done checklist, and the Conflict-Resolution Hierarchy. The law. |
| [`specs/principles-architecture.md`](specs/principles-architecture.md) | When the task touches module boundaries, the lift catalogue (workflows.rs / aggregation.rs / audit.rs), the helper inventory, or any pin-tested invariant. Read the relevant section before editing. |
| [`specs/principles-development.md`](specs/principles-development.md) | When the task involves naming, error handling shape, function-size judgements, helper extraction, or code style. The Rust idioms section is the canonical pattern catalogue. |
| [`specs/principles-mcp.md`](specs/principles-mcp.md) | When the task adds or modifies a tool, changes wire-surface behaviour, touches pagination/caps, modifies error envelopes, or affects the dry-run / cancel / timeout machinery. |
| [`specs/principles-security.md`](specs/principles-security.md) | When the task touches authentication, secrets, logging, error messages exposed to users, audit-trail emission, or any external input validation. |

### Task lifecycle ‚Äî every task, every time

The following ritual is **mandatory** for every task that ships a commit. Skipping any step is a defect ‚Äî call it out in the response rather than silently omitting it.

1. **At task start** ‚Äî open [`specs/constitution.md`](specs/constitution.md) and skim the eight Core Principles + the Definition of Done. If the task touches a specific concern, open the matching `principles-*.md` file from the table above and read the relevant section before writing any code. State briefly which principle docs you consulted.
2. **During the work** ‚Äî when a non-obvious design choice arises, check whether one of the principles or the Helper-First Construction table already answers it. Reach for the existing helper, not a new one. When two principles conflict, apply the Conflict-Resolution Hierarchy from the constitution.
3. **Before committing** ‚Äî walk the Definition of Done checklist in the constitution. Every item must be satisfied; a commit that fails any item is not done. The checklist is the merge gate.
4. **In the response that closes the task** ‚Äî state explicitly which principles were honoured and (if relevant) which pin tests were updated or added. The mention is the audit trail; it makes drift visible.

### Constitution wins on conflict

When this file (CLAUDE.md) and the constitution disagree, **the constitution wins**. CLAUDE.md is the operational guide ‚Äî commands, project structure, known limitations, content-formatting conventions. The constitution and principle docs are the law: design rules, helper-routing invariants, Definition of Done, security constraints. Update the constitution first when a new rule is established; let CLAUDE.md catch up to operational consequences afterwards.

The bundler validates frontmatter (no `<` or `>` characters, description ‚â§ 1024 chars) and writes the skill source as a `<skill-name>/` directory inside the zip ‚Äî the structure claude.ai's Settings ‚Üí Skills upload expects.

### Public vs personal bundle split

The repository ships a **public, generic** skill bundle at `dist/wflow.skill.zip` (sourced from `templates/skills/wflow/`). It carries no user-specific node IDs, no personalised pillars, no cached Workflowy data ‚Äî anyone can download or fork the repo and upload this bundle to their own claude.ai account, then run the in-skill Bootstrap to point it at their own `$SECONDBRAIN_DIR/memory/` and Workflowy account. This is the only artefact the repo ships.

**Personal bundles** (Justin's live skill at `~/.claude/skills/wflow/`, which contains his customised workflows, framework references, etc.) are intentionally NOT auto-built by this repo. Justin uses that source directly for local Claude Code (filesystem reads `~/.claude/skills/wflow/SKILL.md` + the symlinked canonicals at `$SECONDBRAIN_DIR/memory/`). To produce a personal upload for claude.ai web, bundle explicitly:

```bash
scripts/bundle-skill.sh --src ~/.claude/skills/wflow --out ~/some-personal-path/wflow-personal.skill.zip
```

The `dist/wflow.skill.zip` path is reserved for the public bundle. Switching the default away from the personal source on 2026-05-19 was the user's explicit ask: the public-distribution bundle must contain no personal data; the personal data lives in `$SECONDBRAIN_DIR` and is consumed at session start by the in-skill Bootstrap procedure.

### Auto-bundle on skill edit (two pipelines)

A single `PostToolUse` hook (`scripts/auto-bundle-skill.sh`, wired in `.claude/settings.json` matching `Edit|Write|MultiEdit`) classifies every edit and rebuilds whichever bundle(s) the edit feeds:

| Pipeline | Source | Output | Trigger |
|----------|--------|--------|---------|
| **Public** | `templates/skills/wflow/` | `<repo>/dist/wflow.skill.zip` | edits under `templates/skills/wflow/` |
| **Personal** | `~/.claude/skills/wflow/` | `$SECONDBRAIN_DIR/dist/wflow-personal.skill.zip` | edits under `~/.claude/skills/wflow/` OR edits whose realpath matches a file in that directory (covers the symlinked canonicals at `$SECONDBRAIN_DIR/memory/{taxonomy,node_links}.md`) |

Both pipelines run independently. A template edit fires only the public bundle; a personal-skill or canonical edit fires only the personal bundle. The hook:

1. Reads the tool-call payload from stdin and pulls `tool_input.file_path`.
2. Classifies the path against both pipelines (string-prefix for the literal paths; `realpath` equality to catch symlink-traversed edits to the canonicals).
3. For each matching pipeline, calls `scripts/bundle-skill.sh --src <src> --out <out>` and prints a üõé alert line on stderr naming the rebuilt zip path. The alert is what the user sees ‚Äî do NOT also re-state the rebuild in your own response, the hook output is the single source of truth.
4. On bundler failure (frontmatter violation, source error) prints a ‚öÝ STALE warning with the bundler's stderr.

**Why two pipelines:** the user explicitly asked on 2026-05-19 that (a) `dist/wflow.skill.zip` ship only the generic template (no personal data) AND (b) personal-skill edits always publish to Drive so the personal upload artefact stays current. The split prevents personal data from leaking into the public repo while keeping the personal claude.ai upload one click from the Drive web UI.

**Personal bundle requires `$SECONDBRAIN_DIR`** ‚Äî the hook fails loudly when that env var is unset in the subshell, surfacing the same dual-config gap (`.zshrc` vs MCP host config) that `wflow:health_check` warns about.

**Never run `scripts/bundle-skill.sh` by hand after editing either skill source** ‚Äî the hook has already done it. Manual invocations are only required for one-off bundles to non-default paths.

Never ask the user to "remember to re-bundle" ‚Äî the hook removes that obligation entirely. The only outstanding step on the user's side is re-uploading the freshly-bundled zip(s) to claude.ai ‚Üí Settings ‚Üí Skills and starting a fresh session, which the alert message names explicitly. Local Claude Code sees personal-skill changes immediately (reads `~/.claude/skills/wflow/` from disk); the upload is only needed for the web/mobile surfaces.

## Architecture

Rust MCP server for Workflowy content management. Uses `rmcp` 2.2 over stdio transport for Claude Desktop integration.

### Module Structure

- **`src/server/`** ‚Äî MCP tool_router split across `mod.rs` (server struct, helpers, tool_handler!, all 40+ #[tool] handlers, tests) and `params.rs` (parameter struct definitions, ~40 of them). The 2026-05-03 architecture-review file split moved the param struct slab out of mod.rs to make the handler-side navigable. `#[tool]` proc macros register tools; serde + schemars validate inputs via `Parameters<T>` wrapper. Uses `NodeId` newtype for all node ID parameters. Every parameter struct carries `#[serde(deny_unknown_fields)]` so a typo'd field name fails fast with a recorded error instead of silently defaulting to `None`. **The wrapper struct must keep its name `Parameters`**: `rmcp-macros 2.2` discovers a tool's parameter type by matching the literal identifier `Parameters` on the last path segment of the function-arg type (`rmcp-macros/src/common.rs:64`). A wrapper named anything else makes the macro fall back to a hardcoded `{"type": "object", "properties": {}}` schema for every parameter-bearing tool ‚Äî silently strips arguments at the wire. Pinned by `parameter_bearing_tools_publish_non_empty_input_schema_properties`.

  **Helper inventory** (all in `server/mod.rs`):
  - `tool_handler!(self, name, kind, params, body)` ‚Äî the standard wrapper for every non-diagnostic handler. Combines op-log recording + cancel-registry observation + kind-keyed wall-clock budget.
  - `validate_and_resolve(raw) -> Result<String, McpError>` ‚Äî collapses the validate-then-resolve pair every handler taking a node_id needs.
  - `invalidate_for_mutation(&[id, ...])` ‚Äî pre-call cache + name-index invalidation. Every write handler uses it BEFORE the API call so a wrapper-level timeout/cancel can't strand stale data.
  - `truncation_envelope(truncated, limit, reason)` and `with_truncation_envelope(payload, ...)` ‚Äî the four-field JSON envelope for walk-shaped responses (`truncated`, `truncation_limit`, `truncation_reason`, `truncation_recovery_hint`). **Canonical adoption invariant (2026-05-16 sweep):** every walk-shaped tool's JSON payload routes through one of these two helpers ‚Äî `with_truncation_envelope` for fresh-payload merge, `truncation_envelope` for fold-into-existing-Map use. Pinned by `envelope_construction_routes_through_one_helper_no_inline_fields` (forbids inline `"truncation_limit":` JSON keys outside the helpers' own definitions and the test module) AND by `every_walk_tool_emits_full_truncation_envelope_in_json` (residual check that any surviving inline site also carries reason + hint).
  - `tool_error(op, node_id, err)` and `tool_invalid_params(op, node_id, msg)` ‚Äî structured error helpers. The first for operational failures, the second for validation failures. Both produce the same `data` envelope shape so clients see one error model. **Universal contract: every error path emits the structured envelope.** Validation failures route through `tool_invalid_params` (allow-listed only in two helpers ‚Äî `check_node_id` at line ~441 and the short-hash-resolve helper at line ~1207 ‚Äî where the operation name isn't statically known); operational failures route through `tool_error` with no exemptions. Pinned by `handler_body_validation_uses_structured_envelope_not_bare_invalid_params` and `operational_failures_route_through_tool_error_not_bare_internal_error` ‚Äî adding a new bare `McpError::invalid_params(...)` inside a handler body, or any bare `McpError::internal_error(...)`, fails the build before it ships.
  - `workflow_error_to_mcp(operation, node_id, err)` ‚Äî the single canonical translator for `WorkflowyError` returns from `crate::workflows::*` functions. Maps `InvalidInput` ‚Üí `tool_invalid_params`, every other variant ‚Üí `tool_error`. Every handler that delegates to a workflow MUST route the error through this helper rather than hand-writing a `match err { InvalidInput => ‚Ä¶, _ => ‚Ä¶ }` arm ‚Äî pre-2026-05-04 each handler hand-wrote the match and drifted on the operation-name string. Pinned by `workflow_error_translation_routes_through_workflow_error_to_mcp` which forbids `WorkflowyError::InvalidInput` matching anywhere in `server/mod.rs` except inside the helper's own body and the test module.

- **`src/utils/aggregation.rs`** ‚Äî pure aggregation functions shared between MCP handlers and the `wflow-do` CLI: `compute_overdue`, `compute_upcoming`, `compute_recent_changes`, `filter_todos`, `compute_project_summary`, `compute_daily_review`. Take `today` / `now_ms` as parameters rather than reading the system clock. Single source of truth ‚Äî both surfaces call the same function so they can't drift in semantics. **Adoption invariant (2026-05-16):** every list-shaped MCP handler MUST route through the matching aggregation helper. Pinned by `list_shaped_handlers_route_through_aggregation_helpers` which grep-audits the source so a handler that reimplements the date-window / status-filter loop inline fails the build. `compute_project_summary` and `compute_daily_review` returned typed structs (`ProjectSummary` / `DailyReview` with `#[derive(Serialize)]`) rather than `Vec<Value>` because the JSON shape is a single object, not a list ‚Äî the typed shape is the contract.
- **`src/api/client.rs`** ‚Äî Workflowy API client with exponential backoff retry. `get_subtree_recursive()` fetches tree level-by-level via `/nodes?parent_id=` with configurable depth limit (crucial for 250k+ node trees). Returns `SubtreeFetch { nodes, truncated, limit }`; when the `MAX_SUBTREE_NODES` cap (10 000 by default, `defaults.rs`) is hit the flag is surfaced in every tool response so callers can narrow the scope.
- **`src/defaults.rs`** ‚Äî Centralized constants for all magic numbers (cache TTL, retry config, validation limits, tree depth defaults).
- **`src/types.rs`** ‚Äî Core types including `NodeId` newtype with `Deref<str>`, `AsRef<str>`, `PartialEq<String>`, and `JsonSchema` impls. **`Deserialize` is hand-written** (not derived) so the host-encoded literal strings `"null"` / `"undefined"` reject up-front at the parameter boundary instead of routing as opaque IDs that the API layer fails on later. Whitespace-only strings reject too; empty string is preserved as the workspace-root sentinel some handlers (`list_children`, `insert_content`, etc.) use. Surfaced 2026-05-09 by a Claude Desktop session that observed `parent_id="null"` (string, not JSON null) silently routing to contextual destinations across three consecutive `create_node` / `create_mirror` calls. **`WorkflowyNode` serialises sparse (2026-07-21):** absent `Option` fields and the empty `children` list are skipped on output (`skip_serializing_if`) ‚Äî a 10k-node walk payload used to be mostly literal `null`s, paid in tokens and transport on every large read. Absent-vs-null is equivalent to every consumer and the read path (`Option` + `default`) is unaffected. Pinned by `serialised_node_omits_none_fields_and_empty_children`.
- **`src/utils/`** ‚Äî Reusable modules: cache (injectable), date/tag parsing, node paths, subtree collection, rate limiter, job queue, `error_class` (shared `ProximateCause` taxonomy + string‚Üícause classifier used by both the server's `classify_operational_error` and the `wflow-do` CLI's `classify`).
- **`src/cli/`** ‚Äî Standalone CLI binaries (task map generation stub).

### Request Flow

```
MCP tool call ‚Üí Parameters<T> (serde + op_log on failure) ‚Üí tool_handler! (op_log recorder + run_handler: kind-specific budget + cancel) ‚Üí handler body ‚Üí WorkflowyClient ‚Üí retry loop ‚Üí Workflowy API
```

Two recorder points sit on this path:
1. **`Parameters::from_context_part`** records an Err entry to the op
   log when serde rejects the payload ‚Äî covering the path that the
   rmcp framework would otherwise drop before the handler body runs.
   Deserialization runs through `serde_path_to_error::deserialize` so
   the error message carries the offending field path (e.g. ``invalid
   parameters at `.new_parent_id`: invalid type: null, expected a
   string``). Without the path, a host (LLM or MCP client) sending
   `null` for a required `NodeId` saw only "invalid type: null,
   expected a string" with no field name and no way to self-correct;
   the 2026-05-03 `move_node` incident motivated the switch. Pinned
   by `null_required_uuid_field_error_names_the_field`. Failures
   route through `tool_invalid_params` so the wire error carries the
   same `{operation, node_id, hint, proximate_cause, error}` envelope
   as every other validation failure.
2. **`tool_handler!`** (the standard wrapper around every
   non-diagnostic handler body) records the handler's own outcome
   (Ok or Err) AND wraps the body in `run_handler(name, kind, ...)`
   so cancel-registry observation and the kind's wall-clock deadline
   apply uniformly. Diagnostics (`health_check`, `workflowy_status`,
   `cancel_all`, `get_recent_tool_calls`, `build_name_index`) keep
   the older `record_op!` macro because they own short, custom
   budgets that predate the taxonomy.

Together they guarantee every tool call attempt produces exactly one
op-log entry AND that no handler can sit past its kind's budget or
ignore `cancel_all`. Brief 2026-05-02 named the framework-rejection
silence and the unwrapped-write 4-minute hang as the two dominant
debugging black holes; `Parameters` plus `tool_handler!` close
both.

Write operations invalidate the node cache via `self.cache.invalidate_node(id)` (cache is dependency-injected).

### Key Infrastructure

- **Node Cache** (`utils/cache.rs`): Injectable (or global `lazy_static` default), 30s TTL, parking_lot RwLock. O(n) subtree invalidation via parent-children index. **Children-listing cache (2026-07-21):** the cache additionally holds complete, display-ordered children listings keyed by parent id (`""` = workspace root), capped at `defaults::MAX_CACHED_LISTINGS` (4096; overflow purges expired entries, then clears). The `WorkflowyClient` opts in via `.with_node_cache(...)` ‚Äî production wiring (`main.rs`, `wflow-do`) attaches the SAME global instance the server uses for `invalidate_for_mutation`, so the two invalidation layers converge; a bare client (tests) stays fully live. Consult/populate happens in the single children funnel (`get_children_cancellable` / `get_top_level_nodes_cancellable`), so `list_children` AND every walk level benefit; invalidation happens pre-send in the `request_cancellable` write funnel (any non-GET drops the target's listing, the listing containing it, and ‚Äî for creates/moves ‚Äî the body's `parent_id` listing), so no future write path can forget it. `get_node` is deliberately NOT cached (verify-after-write stays live). The diagnostic probe bypasses the funnel and stays live. Covered by the `children_listing_cache` wiremock tests (repeat-read hits upstream once; write invalidates; cache-less client stays live).
- **Rate Limiter** (`utils/rate_limiter.rs`): Token bucket, 10 req/sec, burst 20, **AIMD-adaptive (2026-07-21)**. `stamp_rate_limited` ‚Äî the single site every 429 observation routes through ‚Äî calls `RateLimiter::drain()`, which empties the bucket AND multiplicatively halves the refill rate (floored at 1 req/s), so the post-window resume starts from zero tokens at a slower pace than the one that tripped the limit (the 429-storm oscillation). Each successful upstream response calls `reward()`, recovering the rate additively (0.1 req/s per success) toward ‚Äî never past ‚Äî the configured ceiling. `current_rate()` exposes the adaptive rate for diagnostics. Pinned by `a_429_observation_drains_the_local_token_bucket`, `drain_empties_burst_and_resumes_at_sustained_rate`, and `drain_halves_rate_and_reward_recovers_to_ceiling`.
- **Job Queue** (`utils/job_queue.rs`): Background job lifecycle with TTL cleanup (tokio::spawn). Max 1000 job history.
- **Cancel Registry** (`utils/cancel.rs`): Generation-counter cancellation primitive. `cancel_all` bumps the counter so every outstanding `CancelGuard` returns `is_cancelled = true` at its next checkpoint; guards taken afterwards are fresh.
- **Name Index** (`utils/name_index.rs`): Case-insensitive `name -> [entry]` map plus short-hash ‚Üí UUID maps (12-char URL-suffix and 8-char doc prefix), fed by every subtree walk. Each `NameIndexEntry` also carries the node's `description` (added 2026-07-12 for description-aware `search_tokens`) and `last_modified` (added 2026-07-21, schema `2 ‚Üí 3`, for local incremental queries ‚Äî `wflow-do changed-since <epoch-ms|YYYY-MM-DD> [--root <UUID>]` diffs the index instead of re-walking a subtree on every sync poll; entries with no recorded timestamp are excluded from windows because absence means "not observed"). Both fields are in hand at every ingest site, so populating them costs no extra API calls. `load_from_disk` cleanly discards an unrecognised (older-version) snapshot and lets walks/reindex rebuild it rather than loading silently-missing fields ‚Äî **plan a `wflow-do reindex --timeout-secs 0 --patient` after upgrading across a schema bump**. Pinned by `last_modified_roundtrips_and_changed_since_filters`. Backed by `parking_lot::RwLock`; invalidated per-node on every write. The ingest no-op guard compares name **and** description, so a description-only edit is re-ingested (and marks the index dirty) instead of being skipped with stale text. **Persisted to disk** at `$WORKFLOWY_INDEX_PATH` (unset or empty disables persistence ‚Äî the index then lives only in memory; the repo ships no machine-specific default path so each user wires it through their MCP host config): rehydrated on server startup, checkpointed every 30 s when dirty via merge-then-write-then-rename (see below), and filtered at the serialisation boundary by `WORKFLOWY_INDEX_EXCLUDE_SUBTREES` (see Configuration). **There is no in-process periodic refresher** ‚Äî one walked the workspace root every 30 minutes until 2026-07-17; see the removal note below. Convergence is the scheduled out-of-process `wflow-do reindex --patient` job's responsibility. **Auto-walks on short-hash miss**: `resolve_node_ref` fires a workspace walk with `RESOLVE_WALK_TIMEOUT_MS` budget when a short hash isn't cached; a watcher polls the index every 100 ms and cancels the walk as soon as the target appears. Callers no longer need to run `build_name_index` manually before passing a Workflowy URL fragment. **Disk-refresh-before-walk (2026-07-21):** before firing that walk, the miss path calls `NameIndex::refresh_from_disk_if_changed()` (under `spawn_blocking`) ‚Äî an mtime-guarded merge that adopts entries another writer (the nightly `wflow-do reindex`, another surface) persisted since this process last read the file. A millisecond local read answers the common "the nightly job already covered this node but the long-running server predates it" miss without any API walk; an unchanged file is a no-op. The merge reuses the memory-wins/tombstone-respecting adoption the save path uses, and `save_to_disk`/`load_from_disk` stamp the mtime so a process never re-reads its own writes. Pinned by `refresh_from_disk_adopts_other_writers_entries_and_is_mtime_guarded`. **Create-time seeding (2026-07-21):** `MutationFootprint` carries `created_nodes`, recorded by every creating workflow (`insert_content`, `transaction` creates, `create_mirror`, duplicate/template deep-copies); `apply_footprint` ingests them, so a "copy internal link" short hash of just-created content resolves O(1) ‚Äî pre-fix only the bare `create_node`/`batch_create_nodes` handlers seeded the index. Pinned by `insert_content_seeds_name_index_with_created_nodes`. **Checkpoint mechanics (2026-07-21):** the 30 s saver runs `save_to_disk` under `tokio::task::spawn_blocking` (synchronous fs I/O + cross-process lock + full-index serialise must not stall runtime workers ‚Äî cf. the 2026-06-28 single-vCPU starvation incident) and serialises compact, not pretty (the file is a machine cache; pretty-printing doubled bytes and CPU). Pinned by `index_checkpoint_runs_on_blocking_thread_with_compact_json`.
- **Link Parser** (`utils/link_parser.rs`): `extract_workflowy_short_hash(input) -> Option<String>` is the single canonical extractor turning any user-supplied "link" string into a normalised lowercase hex hash (8/12/32 chars). Handles URL fragment (`/#/<hash>`), `?focusedItem=<hash>` query (wins over the fragment ‚Äî it identifies the inner-most target), shared-URL form (`/s/<slug>/<hash>`), and bare hash/UUID forms (hyphenated or not). Pre-2026-05-19 both `resolve_link` (server) and the `wflow-do resolve-link` CLI inlined the same anti-pattern ‚Äî char-level hex-filter over the whole URL string ‚Äî which silently corrupted on URLs carrying `?focusedItem=‚Ä¶`. Routing invariant pinned by `link_parsing_routes_through_extract_workflowy_short_hash` in `server/mod.rs::tests`, which grep-audits both consumer files for the forbidden pattern.
- **Date Parser** (`utils/date_parser.rs`): Extracts due dates from node text. Priority: `due:YYYY-MM-DD` > `#due-YYYY-MM-DD` > bare date.
- **Tag Parser** (`utils/tag_parser.rs`): Extracts `#tags` and `@mentions` from node text.
- **Node Paths** (`utils/node_paths.rs`): Builds hierarchical display paths by following parent_id chains.
- **Subtree** (`utils/subtree.rs`): Collects all descendants of a node. Todo/completion detection.

### Tree Walk Controls

`get_subtree_with_controls` on the client is the single entry point for every tree walk. All server handlers route through `WorkflowyMcpServer::walk_subtree`, which wraps:

1. A wall-clock deadline (`defaults::SUBTREE_FETCH_TIMEOUT_MS`, 20 s).
2. A `CancelGuard` from the shared `CancelRegistry`.
3. Opportunistic name-index ingestion of every returned node.

`SubtreeFetch` carries `truncation_reason: Option<TruncationReason>` (`NodeLimit`/`Timeout`/`Cancelled`/`SkippedBranches`), `skipped_branches: Vec<String>`, and `elapsed_ms`. Handlers surface these through the `truncation_banner_with_reason` helper and the JSON responses.

**A walk reports coverage, not survival (2026-07-16).** `fetch_descendants` drops a branch whose child fetch errors rather than failing the whole walk over one transient 429 ‚Äî but pre-fix it dropped that branch with nothing but a `warn!` and still returned `complete`, so `complete` meant "did not time out" rather than "covered the subtree". A `wflow-do reindex` that ate a 429 burst reported `1408 nodes ... complete` while whole component menus were missing from the index, and nothing in the response distinguished that from a full walk. Now: each level **retries** its dropped branches once before descending (waiting out an open `retry_after` window, since `request_cancellable` fails fast inside one ‚Äî bounded by the walk's deadline and cancel guard); whatever is still dropped is named in `skipped_branches`, forces `truncated: true`, and ‚Äî when the walk stopped for no other reason ‚Äî sets `truncation_reason: "skipped_branches"`. A stopping reason keeps precedence as the more specific fact, but `skipped_branches` reports alongside it. The recovery hint for this reason **overrides any tool-specific hint** (`SKIPPED_BRANCHES_RECOVERY_HINT`): every other hint points at the persistent name index, which is populated by these same walks and so shares the gap exactly. Pinned by `walk_that_drops_a_branch_reports_skipped_branches_not_complete`, `transient_branch_failure_is_recovered_by_the_retry`, and `skipped_branches_envelope_overrides_even_a_tool_specific_hint`.

Per-level child fetches run via `futures::stream::buffer_unordered(SUBTREE_FETCH_CONCURRENCY)` (5). The rate limiter serialises requests internally, so this parallelism collapses HTTP RTT stalls without exceeding the sustained rate.

### CLI Parity (`wflow-do`)

The `wflow-do` binary at `src/bin/wflow_do.rs` is in **full surface parity** with the MCP tool list above. Every non-diagnostic MCP tool has a matching CLI subcommand routed through the same `WorkflowyClient`. New MCP tools must land with their `wflow-do` subcommand in the same commit. The build-time test `cli_covers_every_non_diagnostic_mcp_tool` enumerates the (mcp-tool ‚Üí cli-subcommand) pairs and fails CI if a tool is added without its CLI counterpart. `convert_markdown` (pure local transform) is intentionally excluded; `cancel_all` and `get_recent_tool_calls` ship as no-op CLI surfaces because the op log only exists in the running MCP server. The CLI also carries local-only subcommands with no MCP counterpart (allowed ‚Äî parity is enforced MCP‚ÜíCLI, not the reverse): `index`, `reindex`, and `changed-since` (2026-07-21; serves "what changed under X since T" from the persisted index with zero API calls, per C-server-020). (`create_mirror` was a stub through 2026-05-04; the failure-report follow-up replaced the stub with a real convention-based implementation, and the `create-mirror` CLI subcommand landed in the same commit.)

### Shared workflow orchestration (`src/workflows.rs`)

Workflow orchestration that used to be duplicated between the MCP server's `#[tool]` handlers and the `wflow-do` CLI subcommands lives in `src/workflows.rs`. Functions there take `&WorkflowyClient` plus typed inputs and a `&WorkflowContext<'_>`, return `(TypedResult, MutationFootprint)`, and are called by both surfaces ‚Äî the MCP handler wraps with `tool_handler!` (cancel + op-log + deadline) and applies the footprint via `apply_footprint`; the CLI propagates errors via `?` and discards the footprint. This is the same pattern `audit::audit_mirrors`, `audit::build_review`, and `utils::aggregation::*` already follow for pure functions; `workflows.rs` extends it to functions that need an API client.

**Foundation types** (all in `src/workflows.rs`):
- `WorkflowContext { cancel: Option<&'a CancelGuard>, deadline: Option<Instant> }` ‚Äî first-class cancel + deadline. MCP fills both; CLI passes `WorkflowContext::default()`.
- `MutationFootprint { invalidated_nodes, invalidated_name_index, created_nodes }` ‚Äî workflows declare which IDs they touched AND which nodes they created (2026-07-21); the MCP wrapper applies invalidation and ingests the created nodes into the name index (short-hash resolution without a walk), the CLI ignores both. Replaces the pre-2026-05-04 hand-written `invalidate_for_mutation(&[id, ...])` arguments at every handler.
- `workflow_error_to_mcp(operation, node_id, err)` (in server/mod.rs) ‚Äî translates `WorkflowyError::InvalidInput` ‚Üí `tool_invalid_params` and every other variant ‚Üí `tool_error`. Replaces the inline `match err { InvalidInput => ‚Ä¶, _ => ‚Ä¶ }` arms.

### Known Limitations

- `bulk_update` supports `complete`, `uncomplete`, `delete`, `add_tag`, `remove_tag`. `complete`/`uncomplete` route through `client.set_completion`, the same code path the single-node `complete_node` tool uses. The wire payload is `POST /nodes/{id}` with `{"completed": true}` or `{"completed": false}` (the read-side `WorkflowyNode::completed` boolean has no serde alias, so the wire field is literally `completed`); pinned by `tests::write_field_names::set_completion_*` so the description‚Üínote failure shape cannot recur on the completion path. The `transaction` tool also accepts `complete` / `uncomplete` ops with rollback to the prior boolean state via `RestoreCompletion`.
- `bulk_tag` (and `bulk_update` with `operation=add_tag`) idempotency uses a whole-tag boundary check via `utils::tag_parser::text_contains_tag`. Pre-2026-05-26 the check was a bare `name.contains("#tag")` substring scan, so a shorter pillar tag was silently skipped whenever a longer existing tag started with the same characters ‚Äî `bulk_tag(tag="lead")` reported `already_tagged` for every node carrying `#leadership`, and the 2026-05-24 pillar-tag backfill missed 38 Lead / 90 Learn / 15 Transform nodes that had to be retagged manually via `transaction` edits. The fix routes both surfaces through `parse_tags` so the match is whole-tag, case-insensitive, and respects the tag-character class (`[A-Za-z0-9_-]`) ‚Äî `#lead` does not match `#leadership` or `#lead-time`. Pinned by `text_contains_tag_matches_whole_tag_not_substring` in `utils::tag_parser::tests`. **2026-06-16 follow-on:** the name-mutation itself (not just the check) is now the shared `utils::tag_parser::add_tag_to_name` / `remove_tag_from_name`, used by the MCP `bulk_tag` handler, `wflow-do bulk-tag`, AND `apply_bulk_op`. Pre-fix the `wflow-do bulk-tag` CLI carried no idempotency check at all ‚Äî it blindly re-appended `#tag`, double-tagging on re-runs ‚Äî because the 2026-05-26 whole-tag fix had only landed on the MCP side. Routing all three sites through the shared helper closes that surface drift by construction.
- `read_batch` dispatches many read ops (`get_node` / `list_children` / `get_subtree`) under a single operations-array wrapper. Use this when the host's MCP transport strips bare-string `node_id` parameters: the operations-array shape inherits the same wire-encoding resilience that `batch_create_nodes` already gives writes. The 2026-05-25 "sweeping a freshly-created node" failure named this gap ‚Äî list_children / get_subtree with bare-string IDs were silently misrouted to workspace root on roughly every second call, leaving the sweep with no clean read path. Operations dispatch with bounded concurrency (`defaults::READ_BATCH_CONCURRENCY`, 4 ‚Äî the shared rate limiter still caps real throughput; changed from strictly-sequential on 2026-07-21 to collapse RTT stalls on multi-read sweeps) and results are emitted in input order; each result carries `{index, op, node_id, ok, data | error}`. Pinned by `read_batch_concurrent_dispatch_preserves_input_order`. `get_subtree` results carry the standard four-field truncation envelope inside `data`. `get_node` requires `node_id`; `get_subtree` requires `node_id` (and accepts `max_depth`, default 5); `list_children` accepts `node_id` optionally (omit / null = workspace root). The CLI subcommand `wflow-do read-batch --input file.json` reads the same wire shape.
- Subtree fetches cap at `defaults::MAX_SUBTREE_NODES` (10 000) **and** at `defaults::SUBTREE_FETCH_TIMEOUT_MS` (20 000 ms). Tools surface a `truncated` flag plus a `truncation_reason` (`node_limit` / `timeout` / `cancelled`) when either budget fires; `duplicate_node`, `create_from_template`, and `bulk_update` (delete) refuse to run against a truncated view.
- **The name index has two writers; `save_to_disk` merges under a cross-process lock rather than clobbering (2026-07-16).** The index file is written by the long-running MCP server (loads at startup, checkpoints every `INDEX_SAVE_INTERVAL_SECS` when dirty) **and** by `wflow-do reindex` (loads, walks, saves once). Both used to serialise their whole in-memory map straight over the file, so each silently discarded every entry the other had added since its own load ‚Äî a window of *hours* for the server. The reported symptom was non-monotonic churn: a subtree present in the file at 15:20 was gone at 16:08, replaced by a different one, even though `ingest` is purely additive and nothing prunes ‚Äî additive semantics cannot explain entries disappearing, but two last-writer-wins savers can. `save_to_disk` is now a **read-merge-write under an exclusive `fs4` advisory lock** on a `.lock` sibling (a sibling, not the index itself, because the save renames a temp file over the index and would swap the locked inode; advisory locks release on handle close, including panic/kill, so a crashed writer cannot wedge the index). The lock spans the whole critical section ‚Äî without it two savers both read, both merge, and the later rename still drops the earlier's additions. **Merge rule:** memory wins for any ID this process knows (it observed that node, so it is at least as fresh); disk-only IDs are adopted; tombstoned IDs are skipped, so the merge that protects additions cannot undo a delete (`invalidate_node` / `clear` record tombstones, cleared on a successful save). An unreadable or wrong-schema file is overwritten rather than erroring ‚Äî refusing to save because someone else wrote a bad file would strand this process's own work. **Residual (documented, not fixed):** a writer that never observes a delete can re-add that node from its own memory on its next save; closing that needs a shared write-ahead log, not a bigger lock, and a stale hit merely 404s on read. Practical consequence: **a `wflow-do reindex` now adds to the index rather than replacing it**, so repeated scoped reindexes accumulate coverage. Pinned by the `multi_writer_merge` test module.
- **The 30-minute in-process index refresher was removed (2026-07-17).** `spawn_index_refresher` walked the *workspace root* every 30 minutes and ingested everything it reached. Two things were wrong with it. First, a root walk ingests indiscriminately, so it silently re-added subtrees that the scheduled reindex is careful to exclude ‚Äî an operator who purged a sensitive subtree from the index got it back within half an hour, invisibly, because nothing in the index records which writer added a node. Second, it could not do its job anyway: a 180 s budget against a 250k-node tree covers a fraction of it, so the walk spent foreground rate-limit budget during working hours to deliver partial coverage the scheduled job already owned. Convergence is now exclusively the out-of-process `wflow-do reindex --patient` job's, which walks *named* roots. The index still rehydrates on startup, still ingests from every walk any tool performs, and still checkpoints every 30 s. `refresh_name_index`, `INDEX_REFRESH_INTERVAL_SECS`, `index_refresh_interval_secs()` and `BACKGROUND_INDEX_WALK_BUDGET_MS` all went with it. Pinned by `no_periodic_root_walk_refresher`. The persistence-time exclusion filter would now catch the leak, but that is defence-in-depth, not licence to restore the thing it defends against.
- **`FetchControls::patient` ‚Äî coverage over latency for unattended walks (2026-07-17).** A walk drops a branch whenever a child fetch returns a non-abort error, and `request_cancellable` fails fast with a synthetic 429 for the whole of an open `retry_after` window. For an interactive caller that is right. For a batch walk it is catastrophic: one real 429 opens a ~50 s window, and every remaining child fetch in that window then fails *in microseconds*, so `buffer_unordered` drains whole levels into the skipped pile at memory speed. A nightly reindex reported `6995 nodes ... complete`-shaped output while dropping **4,603 branches** this way. The once-per-level `retry_skipped_branches` was designed to fix this (it waits the window out) but cannot converge alone: re-fanning thousands of branches at the concurrency ceiling re-trips the limit, the window re-opens mid-flight, and everything still queued fails fast again ‚Äî and there was no second wave. Under `patient`, `recover_skipped_branches` loops waves until a wave stops recovering anything (the no-progress guard: a branch failing for a reason waiting cannot fix, e.g. a 404, must not be retried forever), and fan-out drops to `PATIENT_SUBTREE_FETCH_CONCURRENCY` (2) because the fastest walk is the one that never trips the limit. Default is `false`, so every interactive caller is byte-identical. Exposed as `wflow-do reindex --patient`, which **must** be paired with `--timeout-secs 0`: the retry declines to wait when the wait would outlast the deadline, so a patient walk on a budget quietly stops being patient (the CLI warns if you do this). Pinned by `fetch_controls_are_impatient_by_default`, `walk_fan_out_reads_concurrency_from_controls`, `patient_walk_recovers_a_branch_a_single_retry_wave_would_drop`, and `patient_walk_stops_when_a_wave_recovers_nothing`.
- `find_node` and `search_nodes` refuse to scan from the workspace root when `parent_id` is omitted. Pass `parent_id`, set `allow_root_scan=true` to accept the full walk, or set `use_index=true` after running `build_name_index` to serve from the opportunistic name index. The 2026-05-03 eval run revealed every search scoped under Distillations was timing out at the 20 s walk budget once the subtree grew past the 10 000-node walk cap. `use_index=true` answers in O(1) from the persistent name index without burning any walk budget. **The index match is token-AND over node names AND descriptions (2026-07-12 field report, issues 5 & 6):** every whitespace-delimited query token must appear (in any order) in the entry's `name + " " + description` text. Two pre-2026-07-12 gaps are closed ‚Äî (a) description content was invisible because the index stored only the name (`search_nodes(use_index=true, query="Article 30")` missed a node whose "Article 30" lived in the description); and (b) the matcher lowercased the whole query as one contiguous substring, so a distinctive multi-token query like `"Annex III high-risk regime"` returned nothing even when every token was present. The fast path now routes through `NameIndex::search_tokens` (`find_node`'s `lookup` is deliberately left name-only so its `exact`/`starts_with`/`contains` semantics stay byte-compatible). **Caveat:** a description is only searchable once its node has been walked into the index since the last rebuild, so a live walk stays authoritative for freshly-edited descriptions. Pinned by `search_tokens_matches_multi_token_query_out_of_order`, `search_tokens_matches_description_content`, `save_load_roundtrips_description`, and `description_edit_is_reingested_for_search`. The truncation banner emitted on timeout / node-cap responses names this recovery path explicitly so callers don't have to read the docs. **Index-first everywhere it can be (2026-07-21):** `find_node` and `search_nodes` additionally accept `prefer_index=true` ‚Äî index-first with transparent fallback to the normal scoped walk when the index is empty or has no hits, collapsing the old two-step (use_index, get nothing, re-issue without it) into one call; `tag_search` and `find_backlinks` accept `use_index=true`, scanning the persistent index through the SAME shared predicates their walks use (`node_has_tag`, `node_links_to`), with subtree scoping via parent-chain links (`NameIndex::is_descendant_of`). `find_backlinks` also gained `parent_id` for scoped walks (pre-fix every backlink query was a root walk ‚Äî guaranteed truncation and silently-missed backlinks on a large tree). CLI parity: `--use-index` on `wflow-do search` / `find` / `tag-search` / `backlinks`, each hydrating the persisted index from `$WORKFLOWY_INDEX_PATH` with zero API calls. Pinned by `find_backlinks_use_index_serves_from_index_without_walk`, `tag_search_use_index_matches_whole_tag_from_index`, `find_node_prefer_index_serves_hits_without_walk`, and `search_nodes_prefer_index_falls_back_to_walk_on_index_miss` (C-server-018).
- **JSON-truncation envelope consistency.** Every walk-shaped tool that emits JSON includes the same four fields next to its `"truncation_limit"`: `truncated`, `truncation_limit`, `truncation_reason` (`timeout` / `node_limit` / `cancelled` / null), and `truncation_recovery_hint` (the empty string when not truncated; otherwise the same `TRUNCATION_RECOVERY_HINT` string the markdown banner emits). **Construction must route through one of the two canonical helpers** ‚Äî `with_truncation_envelope(payload, truncated, limit, reason)` for fresh-payload merge, or `obj.extend(truncation_envelope(truncated, limit, reason))` for fold-into-existing-Map use after `serde_json::to_value(&typed)`. Pinned by two tests: (a) `envelope_construction_routes_through_one_helper_no_inline_fields` (2026-05-16) forbids any inline `"truncation_limit":` JSON key outside the helpers' own definitions and the test module ‚Äî the contract becomes enforceable by `cargo build`; (b) `every_walk_tool_emits_full_truncation_envelope_in_json` (residual) verifies any surviving inline site carries reason + hint. The pre-2026-05-16 codebase had ~13 inline emit sites; the sweep collapsed them all to helper calls.
- **`insert_content` creates lines strictly sequentially ‚Äî by design, not oversight (2026-07-21 decision).** Its resume contract ("re-run the remaining lines under `last_inserted_id`") assumes prefix completion: lines `0..created_count` committed, nothing beyond. Parallel sibling creates would leave committed lines beyond a mid-run failure, and the documented resume would then double-write them. For flat sibling batches that want concurrency, use `batch_create_nodes` (per-op status, already pipelined). Pinned by `insert_content_remains_sequential_by_design` (C-server-021).
- `insert_content` caps at `defaults::MAX_INSERT_CONTENT_LINES` (80) per call. The cap was lowered from 200 on 2026-05-04 in response to the failure-report 2026-05-03 session, which observed ‚â•80-line payloads failing at the MCP transport layer with no diagnostic. Above the cap the call returns a typed error with a chunking instruction; chunk to ‚â§80 lines and pass the previous batch's `last_inserted_id` as the next call's `parent_id` to keep the hierarchy stitched together. The previous "soft warn at 80, hard cap at 200" two-tier model collapsed into a single boundary since no observed caller could reliably get above 80 anyway. **`parent_id` is REQUIRED (2026-06-16 hardening): pass an explicit UUID/short-hash, or the empty string `""` for workspace root; omitting or `null` is rejected at the wire with a field-named error.** History: pre-2026-05-04 `insert_content` rejected `null` at the schema layer while peers accepted it (the asymmetry the 2026-05-03 failure-report flagged); 2026-05-04 relaxed it to `Option` (`null`/omit = root, symmetric with `create_node` / `batch_create_nodes`); 2026-06-16 re-tightened ALL FOUR write tools together to required `NodeId` with the `""`-root sentinel, closing the host-stripping/coercion silent-misroute (a stripped or coerced `parent_id` now fails loudly instead of landing at root). Pinned by `write_tools_require_explicit_parent_id_reject_null_and_omit`.
- **`insert_content` reports the committed-count cursor on EVERY failure, not just timeout/cancel (2026-06-17).** Pre-fix the workflow returned three terminal shapes: `Complete`, `Partial { reason: cancelled|timeout }` (carrying `created_count` + `last_inserted_id` as a clean resume cursor), and ‚Äî for a hard mid-batch API error (e.g. a 429 on the 10th of 24 lines) ‚Äî a bare `return Err(e)` that **discarded** the accumulated progress, leaving the caller unable to tell what landed without a separate read. That asymmetry was the write-path report's Recommendation D. The workflow now ALWAYS surfaces its progress via `InsertContentOutcome::Partial`; the new third `PartialReason::Error` discriminates a hard error from the clean cancel/timeout stops, and carries the underlying error string. The MCP handler renders an error-partial as `CallToolResult::error` (`is_error: true`) with `{status:"partial", reason:"error", created_count, total_count, last_inserted_id, stopped_at_line, proximate_cause, retry_after_secs, retryable, hint, error}` ‚Äî the committed-count cursor AND the same classification fields a non-batch failure carries (computed through the shared `classify_operational_error`, so a mid-batch 429 reports `rate_limited` + `retry_after_secs`). Cancel/timeout partials stay success-shaped (expected resume points). **`record_op!` now mirrors `tool_handler!`'s commit-accurate rule** ‚Äî a success-shaped result flagged `is_error:true` records as `Err`, so `per_tool_health.insert_content.ok` cannot tick on a failed batch (extends the 2026-06-01 rule to the `record_op!` path `insert_content` uses). CLI parity: `wflow-do insert` prints the partial JSON and exits non-zero on an error-partial. Pinned by `insert_content_hard_error_returns_partial_with_committed_count` (workflow) and `insert_content_hard_error_partial_surfaces_committed_count_and_rate_limit` (handler envelope + op-log non-commit).
- **Idempotency: `create_node` accepts a best-effort `idempotency_key`; there is no upstream primitive (2026-06-17).** Workflowy's `POST /nodes` assigns the ID server-side (response carries `item_id`) and accepts only `{name, note, parent_id, priority}` ‚Äî no client ID, no `Idempotency-Key` header ‚Äî so true upstream-deduped idempotency is unavailable. `create_node` now takes an optional `idempotency_key`: the server keeps an in-memory map (`utils::idempotency::IdempotencyStore`, TTL `IDEMPOTENCY_TTL_MS` = 10 min, capped at `IDEMPOTENCY_MAX_ENTRIES` = 4096, evict-oldest + prune-expired on access) keyed by that token; a repeated key **replays** the original create's `{item_id, name, parent}` with a reportable `idempotent_replay:` message instead of writing a duplicate. The check runs before any invalidation/write (a replay has zero side effects); the key is recorded only on a confirmed success. **Covered:** retry-after-success and retry-after-fail-before-write (e.g. the in-window 429 fail-fast). **NOT covered (documented, not a bug):** (a) an ambiguous timeout AFTER the POST was sent ‚Äî we never recorded success, so a same-key retry re-issues and double-writes; there the caller must read-back to confirm (a live re-read remains the only proof a write landed, per the 2026-06-01 note); (b) concurrent same-key calls racing the check ‚Äî best-effort, both may write. **`insert_content` / `batch_create_nodes` do not take a key** ‚Äî their structural resume safety (committed-count cursor / per-op `succeeded`+`id`) already makes retry replay-landed-work-free, and a per-line/per-batch key has murky semantics; the key is scoped to the atomic single-node write where the residual double-write risk actually lives. **CLI asymmetry (deliberate):** `wflow-do create` does NOT expose `--idempotency-key` because a one-shot CLI process has no prior call to dedupe against ‚Äî idempotency here is a property of the long-running MCP server, and a no-op flag would imply a guarantee it can't make. Pinned by `create_node_idempotency_key_replays_without_second_write` (wiremock `.expect(1)` proves one upstream write per key) + the `utils::idempotency` store unit tests. If Workflowy later exposes client IDs or an idempotency header, swap the in-memory map for the upstream primitive (it would then also close case (a)).

- `delete_node` and `transaction` delete ops accept an optional `expect_name` name-echo guard (2026-06-16). - `resolve_link` always returns `Ok` with a structured payload ‚Äî on hit `{id, name, description, parent_id, resolved_via}` (where `resolved_via` is one of `cache_hit` / `scoped_walk` / `secondary_attached_index_hit` / `full_uuid_passthrough`); on miss `{resolved: null, short_hash, scope, nodes_walked, elapsed_ms, resolved_via, name_index_size, hint, truncated, truncation_limit, truncation_reason, truncation_recovery_hint}` so the caller branches on `resolved` instead of parsing error strings. - **Resolve-link miss envelope: `resolved_via` discriminator (2026-05-22).** The miss payload carries one of three values: `primary_walk` (this call ran the actual walk and reported its own `nodes_walked`), `secondary_attached` (this call attached as secondary to an in-flight walk for the same scope, polled the name index for the target, and reports `nodes_walked: 0` because the primary did the work), or `walk_error` (the walk errored before completion). Pre-2026-05-22 the secondary path emitted an envelope shape identical to a primary that walked nothing ‚Äî a `nodes_walked: 0, elapsed_ms: 173361` payload looked like a stalled primary when in fact the call had waited 173 seconds behind someone else's walk. The 2026-05-19 user-report named this as the most confusing instrumentation in the response; the `resolved_via` field closes the ambiguity. The miss payload also carries `name_index_size` so the caller can judge whether the persistent index already covers the workspace (in which case a miss likely implies a dead link). Pinned by `resolve_link_miss_payload_carries_primary_walk_resolved_via` and `secondary_marker_released_returns_truncated_false_no_reason`.
- **Resolve-link recovery hints are tool-specific (2026-05-22).** The generic `TRUNCATION_RECOVERY_HINT` ("call `build_name_index`, then re-issue with `use_index=true` on `find_node`/`search_nodes`") is wrong for `resolve_link` ‚Äî those tools search by _name_, not by short hash. `resolve_link` constructs three hint variants matched to the `resolved_via` value: `primary_walk` mentions the persistent index size and the dead-link possibility; `secondary_attached` mentions concurrent walks; `walk_error` mentions `health_check` and rate-limit scoping. The envelope's `truncation_recovery_hint` is also tool-specific via the new `with_truncation_envelope_and_hint` helper. The four-field envelope shape stays identical so `every_walk_tool_emits_full_truncation_envelope_in_json` still holds.
- **Secondary marker-released branch reports `truncated: false` (2026-05-22).** When a secondary observes the primary release the scope marker without resolving the target, the secondary now returns `truncated: false, truncation_reason: None` because _the secondary didn't walk_. Pre-fix this branch returned `truncated: true, truncation_reason: None`, which (a) violated the four-field-envelope contract (Principle 3: every truncated response carries a reason) and (b) misled callers into thinking _their_ call hit a limit. The handler distinguishes "secondary attached, target not in index" from "primary walked, target not in index" via the `resolved_via` field rather than via `truncated`. The secondary's own deadline branch still reports `truncated: true, reason: Timeout` because that path truly did hit a bound.
- **`RESOLVE_WALK_TIMEOUT_MS = SUBTREE_FETCH_TIMEOUT_MS` (20 s; lowered from 180 s on 2026-05-22).** The on-demand resolve walk now shares the standard subtree budget. Pre-fix the resolve walk had its own 180 s budget on the theory that the user was _waiting_ for the resolution and we should give the walk every chance to succeed; the 2026-05-19 user-report showed that on 56k-node workspaces this budget left the caller hanging for nearly three minutes per cold-cache miss (level-by-level child fetches at the rate-limit ceiling cannot drain that tree in three minutes anyway). The 20 s budget converts the failure mode from "three minutes, then null" to "twenty seconds, then null" so the caller can branch on the miss envelope and try a scoped retry instead of losing minutes per attempt. Exhaustive coverage is the scheduled `wflow-do reindex --patient` job's business, not an interactive caller's (the in-process background refresher that used to own it was removed 2026-07-17). Pinned by `resolve_walk_budget_aligns_with_subtree_fetch_budget` (in addition to the existing `resolve_walk_budget_leaves_mcp_transport_margin`).
- **CLI is a facade over the lifted workflow (2026-05-22).** The `wflow-do resolve-link` subcommand now routes through `crate::workflows::resolve_link_via_walk_and_scan` ‚Äî the same orchestration the MCP `resolve_link` handler uses to walk a scope and look for a short-hash. Pinned by `cli_resolve_link_emits_full_envelope_on_miss`. **CLI uses the persistent index too (2026-07-21):** when `$WORKFLOWY_INDEX_PATH` is configured, `wflow-do resolve-link` hydrates the index from disk and preflights the short hash (a hit costs one verifying `get_node`; a stale entry falls through to the walk), and after any walk it ingests the walked nodes and merge-saves them back ‚Äî so every miss-walk permanently extends coverage instead of being discarded, and the miss envelope's `name_index_size` reflects the hydrated index. Pre-fix the CLI threw its walks away and re-walked the same region on the next miss.
- **Resolve-walk single-flight per scope.** `walk_for_short_hash` and `walk_for_short_hash_scoped` both route through `walk_for_short_hash_inner`, which maintains an `inflight_resolve_walk_scopes: HashSet<Option<String>>` keyed by parent UUID (None = workspace root). Pinned by `concurrent_resolve_walks_share_one_walk_per_scope`.

- **Diagnostic probe bypasses the rate limiter.** `health_check` and `workflowy_status` issue their probe via `WorkflowyClient::probe_top_level`, which skips both the rate-limit acquire and the retry loop. Pre-2026-05-19 the probe went through `get_top_level_nodes_cancellable`, which queued behind every pending walk acquire; on a saturated limiter the 5-second probe budget elapsed in the queue without ever issuing a network round-trip, producing `"two attempts failed: Timeout | Timeout"` with no diagnostic distinction between "upstream down" and "local queue wedged". The bypass is single-shot per probe (~one request per minute in degraded state) so the upstream impact is negligible. Pinned by `probe_upstream_with_retry_uses_probe_top_level_not_throttled_path`.
- **Diagnostic probe is single-flighted (2026-05-26).** `probe_upstream_with_retry` takes `self.probe_lock.lock().await` at the top, so concurrent `health_check` / `workflowy_status` invocations serialise onto one in-flight HTTP probe. - **Diagnostic probe is suppressed inside an open retry_after window (2026-05-26).** `WorkflowyClient` tracks `last_429_unix_ms` + `last_retry_after_ms` and exposes them via `rate_limit_posture()`. When `probe_top_level` is called inside the window (`in_retry_window: true`), it short-circuits and returns a synthetic `ApiError { status: 429, message: "{retry_after: N}" }` from the cached posture **without issuing an HTTP request**. The 2026-05-26 incident: a Claude session diagnosing rate-limit pressure observed `retry_after` resetting on every `workflowy_status` poll because each probe consumed a token ‚Äî measurement was perturbing the system. Suppression converts the polling anti-pattern into a free no-op until the upstream window genuinely clears. `probe_upstream_with_retry` also skips its second attempt on a 429 first-attempt (same reason: a retry inside the window would just consume more quota with no new information). - **`workflowy_status.degraded_kind` classifies the cause of `degraded`.** Possible values: `null` (status is ok), `"rate_limited"` (recent 429 ‚Äî `retry_after_remaining_ms` carries the open window in the same payload), `"auth"` (recent 401/403), `"cancelled"` (probe cancelled mid-flight), `"local_queue_wedged"` (probe timed out AND in-flight walks > 0 ‚Äî local rate limiter is the bottleneck), `"upstream_unreachable"` (probe timed out with no in-flight work ‚Äî upstream is down), or `"upstream_error"` (any other non-success probe outcome). - **The request path fails fast inside an open `retry_after` window (2026-06-01).** `request_cancellable` ‚Äî the central path every read and write routes through ‚Äî checks `rate_limit_posture().in_retry_window` BEFORE acquiring a rate-limiter token or entering the retry loop, and short-circuits with the same synthetic 429 (`status: 429`, body `{"error": ..., "retry_after": N}`, where `N` is the _remaining_ seconds) that the diagnostic probe returns. Pre-fix, a call issued inside the window queued behind the rate limiter and held for the full ~4-minute MCP transport timeout before returning "no result" ‚Äî indistinguishable from a crashed server and the single biggest time-sink in a bulk write session (the 2026-06-01 429-storm). The short-circuit converts that into a sub-millisecond structured rate-limit error the caller can branch on (wait `retry_after`, then retry). Both suppression sites ‚Äî the probe (`probe_top_level`, which bypasses the rate limiter via `try_request_cancellable`) and the request path ‚Äî build the error through the shared `WorkflowyClient::rate_limit_window_error` helper so the wire shape cannot drift; `classify_degraded_kind` routes both to `rate_limited`. The probe and request paths don't double-fire (the probe calls `try_request_cancellable` directly). The `wflow-do` CLI inherits the same fail-fast for free (same `WorkflowyClient`). Pinned by `request_path_fails_fast_inside_retry_window` (latency assertion < 50 ms against an unreachable upstream) and `request_cancellable_short_circuits_inside_retry_window` (grep: the preflight precedes the retry loop).
- **A 429 reaching the per-call error envelope classifies as `rate_limited`, not `unknown` (2026-06-17).** `tool_error` classifies by string-matching the error, and pre-fix it had branches for 404 / cancelled / timeout / 5xx / 401-403 / lock / cache but **no 429 branch** ‚Äî so a rate-limited write (the synthetic in-window short-circuit from `request_cancellable`, or a real upstream 429 that exhausts retries) fell through to `proximate_cause: "unknown"` with the hint "see data field for details", burying the `retry_after`. This was the per-call half of the write-path failure report's "writes fail with no diagnostic": the *status* path (`classify_degraded_kind`) already treated 429 as first-class `rate_limited`, but the per-call envelope disagreed. Fix: a 429/`rate limit` branch placed FIRST in the classifier (so no later match shadows it) ‚Üí `ProximateCause::RateLimited`. Two new typed envelope fields land with it: `retryable` (derived from the cause ‚Äî true for recoverable transients incl. rate-limit/timeout/upstream/lock/propagation-404/cancelled, false for auth/validation/unknown ‚Äî so it cannot drift from the classification) and `retry_after_secs` (the 429 `retry_after` lifted from the raw body into a typed number, mirroring `workflowy_status.retry_after_remaining_ms`; `null` on every non-rate-limited path). `tool_invalid_params` carries `retryable: false` for envelope symmetry. The report named `retryable` + `retry_after` as the two fields that turn blind retrying into actionable wait-and-retry. NOTE: this codebase has NO separate write pool/limiter ‚Äî reads and writes share one `WorkflowyClient` and one rate limiter ‚Äî so the report's read-OK/write-fail asymmetry is upstream POST-vs-GET rate-limiting, not a wedged local write path; no circuit breaker was added. Pinned by `tool_error_rate_limited_envelope_carries_retry_after_and_retryable`, `parse_retry_after_secs_extracts_from_both_shapes`, and the extended `tool_error_proximate_cause_classification_covers_every_branch`.
- **A rolled-back `transaction` is a non-commit: `is_error: true` + recorded as `Err` (2026-06-01).** `per_tool_health.<tool>.ok` counts the op log's `OpStatus::Ok` entries, which previously meant only "the handler returned `Ok`" ‚Äî and `run_transaction` returns `Ok(TransactionOutcome::RolledBack {..})` on rollback, so the ok counter advanced even though nothing durably committed. The 2026-06-01 429-storm exposed the trap: transactions rolling back under a 429 still ticked the ok counter, so the assistant believed writes had landed and re-applied work that had not. Fix: the `transaction` handler now renders a `RolledBack` outcome as `CallToolResult::error` (`is_error: true`) ‚Äî the caller still receives the full rollback envelope as content, but MCP clients see the operation as unsuccessful AND `tool_handler!` records the op-log entry as `Err`. The macro change is general: any handler returning a success-shaped result flagged `is_error: true` is recorded via `finish_err(op_err_summary(r))`, so `per_tool_health.<tool>.ok` is commit-accurate, not receipt-accurate. The handler still returns `Ok` at the Rust level (rollback is part of the contract, not a protocol error), so `transaction_rejects_unknown_op_kind`'s Ok-with-rolled-back-payload expectation holds. CLI parity unaffected ‚Äî `wflow-do transaction` calls the workflow directly and prints the unchanged `RolledBack` outcome. **The only reliable confirmation a write landed remains a live re-read of the node**; the commit-accurate counter removes the false positive, it does not replace verify-after-write. Pinned by `rolled_back_transaction_records_as_noncommit_not_ok` and `tool_handler_records_is_error_true_results_as_finish_err`.
- **`ProximateCause` + the string‚Üícause classifier live in `utils::error_class`, shared by the server and the CLI (2026-06-24).** Pre-fix the enum + classification logic lived in `server/mod.rs` (private), and `wflow-do`'s `classify()` carried a hand-copied branch chain that had already drifted: it lacked the `rate_limited` (429) branch the server gained 2026-06-17, so a rate-limited CLI error printed `proximate_cause: unknown`. The cause taxonomy, `ProximateCause::from_error_message` (429 branch first, cannot be shadowed), and `is_retryable` now live in one module; `server::classify_operational_error` calls them and maps the cause to the MCP-specific `ErrorCode`+hint locally, and the CLI's `classify()` is a one-liner over `from_error_message().as_str()`. `server/mod.rs` re-exports `ProximateCause` so every existing reference is unchanged. The classification can no longer diverge between surfaces by construction. Pinned by `utils::error_class::tests::*` (branch coverage incl. the 429-first ordering) and the CLI's extended `classify_covers_known_branches` (now asserts the rate_limited branch).
- **Every UUID-bearing single-node tool emits `scope_resolved` (2026-06-24 residual).** The `scope_resolved:` audit signal (originally on `list_children` / `create_node` / the create-family) now also lands on `get_node`, `edit_node`, `delete_node`, `move_node`, and `complete_node` ‚Äî text-returning tools prefix `scope_resolved: <label>`, `get_node`'s JSON payload carries a `scope_resolved` key, and `move_node` names BOTH resolved scopes (`node=‚Ä¶; new_parent=‚Ä¶`) since both params are host-coercion-vulnerable. This closes the read/edit half of the 2026-05-30 observation that single-node `edit_node`/`get_node` were exactly where a host-coerced `node_id` silently misrouted (the operations-array tools were already 100% reliable). It is defence-in-depth visibility ‚Äî a host that coerces `null` into a *plausible real UUID* before the deserializer remains unpreventable server-side, but the caller can now always see what the server actually targeted. Pinned by the extended `every_scoped_tool_emits_scope_resolved_in_response` (now includes the five single-node tools) and the updated `complete_node_*` response-shape assertions.
- `health_check` calls `GET /nodes` (top-level only) with a 5-second budget via the rate-limiter-bypass probe path; it is safe to use as a liveness probe regardless of tree size or rate-limit pressure.
- `list_children` accepts both `node_id` and `parent_id` for the parent-to-list-children-of, since every other tool in this server that scopes to a parent uses `parent_id`. Both names reach the handler with the same semantics. Any third name fails the `deny_unknown_fields` check with a typed `unknown field` error.
- **Child listings are sorted into Workflowy display order (2026-07-12 field report, issue 3).** The `/nodes?parent_id=` endpoint returns children in an internal/creation order, **not** the outline's display order ‚Äî and `get_subtree` additionally interleaved deeper levels via `buffer_unordered`, so `list_children` and `get_subtree` could report different "first" children for the same parent and neither matched the UI. Every children listing now routes through the single funnel `get_children_cancellable` (and the workspace-root `get_top_level_nodes_cancellable`), which sorts by `priority` ascending via `WorkflowyClient::sort_children_by_priority`. Ascending priority = top-of-outline was confirmed empirically (a live top-level read: the item Workflowy renders first carried the lowest priority value) and is consistent with `reorder_nodes`' proven `priority=0`‚Üíhead semantics. The sort is stable, so ties and priority-less nodes (`None`, sorted to the head) keep their upstream order rather than shuffling. Result: `list_children` and `get_subtree` agree by construction (both funnel through the same call) AND both reflect the UI, so verify-after-write is reliable. Pinned by `sort_children_by_priority_ascending_none_first_stable`.
- **`insert_content` assigns explicit ascending priorities so an ordered page is not reversed (2026-07-12 field report, issue 2a).** Pre-fix every per-line create passed `priority=None`, ceding order entirely to Workflowy's placement of priority-less creates ‚Äî which lands each new sibling at the head, reversing an ordered list (the reversal was also non-deterministic across sessions). Each line now creates with `priority = idx` (its 0-based input position, bounded by the 80-line cap), so input order maps onto ascending priority and ‚Äî composed with the ascending read-sort above ‚Äî a page inserted top-to-bottom reads back top-to-bottom. Workflowy rescales priority server-side but preserves *relative* order (see the priority note below), so only the monotonic sequence matters, not the absolute value. This is **best-effort**: if the upstream declines to honour create-time priority, order falls back to upstream placement and `reorder_nodes` (reverse-`priority=0`, the guaranteed primitive) remains the deterministic escape hatch. Pinned by `insert_content_assigns_ascending_priorities_per_line` (wiremock `.expect(1)` per `{name, priority}` pair).
- **Upstream priority artifacts we do NOT control (2026-07-12 field report, issues 2b & 2c).** These are Workflowy-server behaviours the client faithfully passes through; the fix is to route ordering through `reorder_nodes`, not to fight them: (2b) a same-parent `move_node` sends `priority` on the wire (`client.move_node` attaches it unconditionally; no same-parent special-casing), but Workflowy's `/move` endpoint no-ops/renormalises priority when the parent is unchanged, so a bare same-parent reorder via `move_node` is unreliable ‚Äî `reorder_nodes` (which re-plants each node at `priority=0` in reverse) is the reliable in-place reorder. (2c) Priority values are rescaled on write (set `1000, 900, ‚Ä¶` and read back `700, 600, ‚Ä¶`); relative order is preserved but the absolute key is not, so callers must rely on a relative sequence and never a stored priority value. The read type is `Option<i64>`, write type `Option<i32>` ‚Äî a benign asymmetry; the server neither clamps nor rescales.
- **`resolve_link`/id handling: the claude.ai host strips bare-string id params before the connector sees them (2026-07-12 field report, issue 1 ‚Äî NOT server-fixable).** Confirmed in production and already the standing rationale for `read_batch` / `transaction`: on the claude.ai custom-connector surface, a top-level bare-string `node_id` is dropped by the *host* before the request reaches the Fly connector (which is a thin transport shim over this same Rust server and transforms no params); the same id nested inside an operations array (`read_batch`, `batch_create_nodes`, `transaction`) survives. No server code can recover an id that never arrived, so the mitigation stays: route scoped reads through `read_batch` and writes through the operations-array tools, and rely on the required-`parent_id` / `expect_name` / `scope_resolved` hardening. A genuine fix requires the host-side serialization to be corrected upstream. Documented in `docs/REMOTE-CONNECTOR.md`.
- **Bare `{"error":"Error occurred during tool execution","request_id":...}` failures originate above our handler (2026-07-12 field report, issue 4 ‚Äî NOT server-fixable in-handler).** That string exists nowhere in this repo; every failure path reachable inside our code emits a structured body (the `insert_content` partial-success cursor with `created_count`/`last_inserted_id`, or the `tool_error`/`tool_invalid_params` envelope). The bare shape with a `request_id` (a field this server never emits) is the rmcp framework or the Fly/claude.ai transport wrapping a torn or timed-out connection ‚Äî i.e. the handler never returned (e.g. the ~4-min MCP transport hard-timeout fired before our inline deadline produced its partial). Recovery when it appears: read-back to confirm what landed before retrying (a live re-read remains the only proof a write committed). The server-side guarantee ‚Äî one structured op-log entry and one partial cursor per in-code failure ‚Äî is already in place and unchanged.
- **Large `get_subtree` results spilled to a file are claude.ai host rendering, not server truncation (2026-07-12 field report, issue 7).** The server applies no byte-size cap to tool payloads ‚Äî only node-count (10 000) / result-count / wall-clock (20 s) truncation, each surfaced honestly via the four-field envelope. A completed sub-cap `get_subtree` serialises its full JSON; the host decides to spill an oversized result to a file. There is no cursor/continuation pagination (the Workflowy REST API has no server-side child paging to hang one on); narrow the scope (`root_id`, shallower `max_depth`) or use `read_batch` per-branch to stay under the budget.
- `degraded` self-clears on recovery: once the failing tool returns OK, `workflowy_status.last_failure` and the `DEGRADED` warning that gates `create_node` both clear together via `OpLog::last_unrecovered_failure`. A success on a _different_ tool does NOT clear another tool's failure warning.
- `create_mirror` implements a documented `mirror_of:` / `canonical_of:` note convention, NOT native Workflowy mirrors (the public REST API does not expose mirror creation). Edits to the canonical do **not** propagate to the mirror ‚Äî the link is structural and human-curated, audited via `audit_mirrors`. Optional `pillar` parameter writes a `canonical_of: <pillar>` marker to the canonical when it lacks one; existing markers are never overwritten. Mirror-of-self is rejected up-front. The orchestration lives in `crate::workflows::create_mirror_via_convention` and is shared with the `wflow-do create-mirror` CLI.
- `audit_mirrors` separates the **walk scope** (which subtree is traversed to find mirrors) from the **canonical-resolution scope** (where `mirror_of:` UUIDs are looked up to decide whether they resolve). Cross-pillar mirroring is the standard Mirror Discipline pattern, so a mirror's canonical living in a different pillar is the normal case rather than a fault. Two behaviours fall out of that split:
  - **Chunked walk** (default when `root_id` is omitted): the handler lists the root's direct children and walks each as its own subtree with `defaults::SUBTREE_FETCH_TIMEOUT_MS` budget, then unions the node sets. The Distillations subtree exceeds the 10 000-node walk cap, so a single unchunked walk against the default root times out at the root and reports zero findings. Chunking sidesteps the cap because each pillar fits comfortably. Pass `chunked=false` to opt out (e.g. when scoping to a small subtree where the cap is irrelevant); pass `chunked=true` with an explicit `root_id` to force chunking under any scope. Per-chunk envelope (`id`, `name`, `scanned`, `truncated`, `truncation_reason`) is surfaced under `chunks` so callers can see which pillar timed out.
  - **Widened canonical resolution** (default `cross_scope_resolve=true`): for every `mirror_of:` UUID encountered in scope that the walk itself didn't resolve, the MCP handler consults the persistent name index via `NameIndex::lookup_entry_by_id` (O(1), ~55k entries in production). When the canonical is found there, it is added to the `external_canonicals` map passed to `audit::audit_mirrors_with_external` and the mirror classifies OK (or DRIFTED if names diverge). Pre-2026-05-16 the audit conflated the two scopes and emitted false-positive BROKEN findings for every legitimate cross-pillar mirror ‚Äî the symptom the 2026-05-16 weekly-synthesis report named. The CLI (`wflow-do audit-mirrors`) implements the same widening by issuing one `get_node` per unresolved UUID (rate-limited via the client). Pass `cross_scope_resolve=false` to restore the legacy in-scope-only classifier. The ORPHAN check is skipped for external canonicals when the resolver path doesn't carry the canonical's description (the name index doesn't store descriptions, so the handler treats marker-presence as unknown rather than absent ‚Äî only `has_canonical_marker=Some(false)` triggers ORPHAN, never `None`).
  - **`mirror_of:` resolution recognises both short-hash shapes, not just one (2026-07-01).** `audit.rs`'s `id_match` helper (shared by the in-scope lookup and the external-map lookup) matched a full UUID against a `mirror_of:` value via `ends_with` only ‚Äî the 12-char URL-suffix short hash. `link_parser.rs` documents a second first-class short-hash shape, the 8-char doc-form prefix, which `id_match` never matched because a leading-characters marker never satisfies a suffix check. Surfaced by a full-pillar `audit_mirrors` sweep (Decide pillar) that reported two BROKEN findings for `mirror_of:8192d7b6` even though the target node was in the same scan. Fixed by adding a length-guarded (`>= 8` chars) `starts_with` branch alongside the existing `ends_with` one, matching the shapes `link_parser.rs` already recognises. Pinned by `mirror_of_8_char_doc_prefix_hash_resolves_in_scope` and `mirror_of_8_char_doc_prefix_hash_resolves_via_external_map`.
  - **DRIFTED name comparison ignores trailing `#tags` (2026-07-01).** The DRIFTED check compared a mirror's full name (claim text + trailing tags) against the canonical's, so a mirror carrying pillar-appropriate tags that legitimately differ from the canonical's tags ‚Äî the normal case, since mirroring the same claim into six pillars means six different tag sets by design ‚Äî always false-flagged as DRIFTED even with byte-identical claim text. Surfaced by a full 7-pillar `audit_mirrors` sweep: 10 of 15 DRIFTED findings turned out to be tag-only divergence, confirmed by the user as expected pillar-contextual tagging, not drift. Fixed via a new `strip_trailing_tags` helper (strips one-or-more trailing `#tag` tokens via regex) applied to both names before the containment check, at both the in-scope and external-map comparison sites. Pinned by `mirror_with_identical_claim_but_different_pillar_tags_does_not_drift` and `mirror_with_genuinely_different_claim_still_drifts_regardless_of_tags` (the latter proves a real wording divergence still classifies DRIFTED even when tags also differ, so the fix doesn't mask genuine drift).

## Configuration

Environment variables loaded from `.env` via dotenvy (`src/config.rs`):
- `WORKFLOWY_API_KEY` (required)
- `WORKFLOWY_INDEX_PATH` (optional) ‚Äî disk path for the persistent name index. Unset or empty disables persistence. The repository ships no machine-specific default; each user (or MCP host config) wires the path explicitly.
- `WORKFLOWY_INDEX_EXCLUDE_SUBTREES` (optional) ‚Äî comma-separated full UUIDs and/or 12-char URL short hashes whose subtrees must never be **written to the persistent index**. Unset or empty excludes nothing. The rule is deliberately "may enter memory, may never reach disk": a walk stays free to traverse an excluded subtree (a live session resolving a hash under it still needs an answer, and the in-memory index dies with the process), but the on-disk file is a durable artefact read by other tools and must not carry it. The filter therefore sits at the serialisation boundary ‚Äî `NameIndex::snapshot()`, the sole caller of which is `save_to_disk` ‚Äî so **every** writer passes through it: the server's 30 s checkpoint and an out-of-process `wflow-do reindex` alike. Exclusion is transitive (root + all descendants) and seeds on both a node's own id **and** its `parent_id`, because the excluded root is normally *absent* from the index once purged; seeding on id alone would find nothing to walk from and pass the whole subtree through ‚Äî failing open precisely where it must not (observed 2026-07-17). A token that is neither a 36-char UUID nor a 12-char short hash is dropped **with a warning**, since a silently-ignored exclusion token is a privacy leak rather than a cosmetic bug. Pinned by the `excluded_subtrees` test module, including `excludes_descendants_even_when_the_excluded_root_is_absent` (proved to go red against its own defect) and `persisted_snapshot_is_the_only_serialisation_path`.
- `SECONDBRAIN_DIR` (optional) ‚Äî root of the user's operational `secondBrain` directory (drafts, session logs, briefs, memory). The `review` tool's bucket-d session-log scan and the `wflow-do index` default output path read from `$SECONDBRAIN_DIR/session-logs/`. Unset or empty disables those features (graceful skip). **Write paths fail loud on a stale path:** `defaults::secondbrain_dir_checked()` returns `Ok(None)` when unset (feature disabled) but `Err` (naming the resolved path) when the env var is set yet the directory or its `memory/` subdir is missing ‚Äî so `wflow-do index` refuses to write `INDEX.md` to an unprovisioned location instead of creating files where nobody expects them (2026-06-02 relocation hazard). Read paths that already skip on `None` keep using the bare `secondbrain_dir()` accessor. Pinned by `secondbrain_dir_checked_distinguishes_unset_missing_and_present`.
- `WORKFLOWY_REVIEW_ROOT` (optional) ‚Äî default root node for `review` and `audit_mirrors` when `root_id` is omitted (read via `defaults::default_review_root()`, shared by the MCP handlers and the `wflow-do` CLI). **No hardcoded fallback** (constitution: "no machine-specific IDs"); when unset, both tools return a typed invalid-params error asking for an explicit `root_id`. Replaced the former hardcoded `DEFAULT_REVIEW_ROOT` const.

## Templates and setup

The repo ships generic templates so a fresh user (or LLM agent bootstrapping one) can stand up the same workflow without inheriting the original author's specific node IDs:

- `docs/SETUP.md` ‚Äî step-by-step guide for an LLM to install the MCP and provision the user's secondBrain directory.
- `templates/skills/wflow/SKILL.md` ‚Äî generic copy of the wflow skill with no user-specific node IDs.
- `templates/secondbrain/` ‚Äî skeleton of the operational secondBrain directory (README, memory schema, drafts/session-logs/briefs subdirs).

User-specific data (cached node IDs, drafts, session logs, briefs) lives in whatever path the user sets via `$SECONDBRAIN_DIR` ‚Äî never in this repo. The repo carries no opinion about where on disk that lives.
